# Platty의 동작 원리

> 명령어를 실행하기 전에 *Platty가 무엇이고 왜 신뢰할 수 있는지*를 먼저
> 이해하고 싶다면 이 문서부터 읽어 보세요.
> 바로 설치할 준비가 되셨나요? [사용 가이드](usage-guide.md)로 이동하세요.

---

## Platty란 무엇인가

**Platty는 브라운필드에서 AI 코딩이 통하게 만듭니다.** AI 에이전트는 빈
저장소에선 훌륭하지만 실제 코드베이스에선 무너집니다 — 기존 시스템을 모르기
때문입니다. Platty는 여러분의 실제 코드베이스에서 살아있는 **소스 오브
트루스(SOT) 스펙**을 역추출합니다 — 라우트, 데이터 모델, 데이터베이스 접근,
API 호출, 이벤트, 잡, 외부 서비스까지 — 그래서 에이전트(그리고 여러분)가
시스템이 *실제로* 동작하는 방식을 기준으로 계획·스펙 작성·구현을 할 수
있습니다. 이미 가지고 있는 코드에서 시작하는 스펙 주도 개발(SDD)이며, 같은
스펙이 검색하고 신뢰할 수 있는 기술 문서와 비즈니스 문서도 만들어 냅니다.

핵심 아이디어는 이렇습니다. 코드가 바뀌는 순간 낡아 버리는 손으로 쓴 문서를
사람들에게 요구하는 대신, Platty는 문서를 **소스 그 자체로부터** 도출합니다.

---

## 두 단계

Platty는 두 단계로 동작합니다. 내부 구조를 이해할 필요는 없습니다 — 이 두 가지
개념만 알면 됩니다:

```
   ┌─────────────────────┐        ┌─────────────────────┐
   │  1. Static analysis │   →    │   2. LLM analysis   │
   │  (reads your code)  │        │  (writes the docs)  │
   └─────────────────────┘        └─────────────────────┘
        runs on your              uses the AI provider
        own machine               you choose
```

**1. 정적 분석(Static analysis).**
Platty는 여러분의 소스 코드를 읽습니다 — 절대 실행하지 않습니다 — 그리고 코드가
무엇을 하는지를 소스에 근거한 지도로 구축합니다. 이것은 순수한 분석입니다:
파일을 파싱하고, 임포트와 호출을 따라가며, 구체적인 증거를 기록합니다(어떤
라우트가 어떤 테이블을 읽는지, 어떤 API를 호출하는지, 어떤 큐에 발행하는지, 어떤
벤더를 사용하는지).

**2. LLM 분석(LLM analysis).**
Platty는 그 지도를 AI 모델에 넘기고, 모델은 기술 문서와 비즈니스 문서를
작성합니다. 모든 문장이 정적 지도에 근거하기 때문에, 각 주장은 추측이 아니라
실제 소스 코드로 거슬러 올라가 추적할 수 있습니다.

일상적으로 보면 이것은 두 개의 명령어일 뿐입니다 — `platty analyze`(정적 분석)와
`platty generate-docs run`(LLM 문서화). [사용 가이드](usage-guide.md)를
참고하세요.

## 핵심 원칙: 의미를 절대 지어내지 않습니다

Platty의 가장 중요한 규칙은 **보수적 정직함**입니다: 소스가 뒷받침할 수 없는
연결을 절대 만들어 내지 않습니다.

Platty가 두 코드 조각이 연결되어 있음을 증명할 수 없을 때, 링크를 **지어내지
않습니다**. 대신 미해결 증거를 그것이 미해결로 남은 이유와 함께 보존합니다 —
그래서 그 간극은 그럴듯하게 들리는 추측으로 덮이지 않고 여러분의 문서에 드러나
보입니다.

이것이 Platty의 출력이 신뢰할 수 있는 이유입니다: 누락된 엣지는 누락된 엣지로
기록되며, 숨겨지지 않습니다.

## 로컬 우선: 신뢰할 수 있는 이유

**여러분의 코드와 분석은 여러분의 컴퓨터에 머무릅니다.**

- **정적 분석은 완전히 로컬에서 실행됩니다.** 여러분의 소스는 절대 업로드되지
  않으며, Platty는 여러분의 코드를 **절대 실행하지 않습니다**.
- **여러분의 데이터는 로컬 데이터베이스에 저장됩니다.** 추출된 지도 / 소스 오브
  트루스는 `~/.platty` 아래의 로컬 SQLite 데이터베이스에 저장됩니다 — 어떠한
  클라우드에도 저장되지 않습니다.
- **AI 단계는 여러분이 통제합니다.** 오직 LLM 문서화 단계만 AI 모델과 통신하며,
  **여러분이 선택한 프로바이더와 자격 증명**을 사용합니다. 무엇을 어디로 보낼지
  여러분이 결정합니다.

Platty는 오픈 소스가 아닌 독점 소프트웨어이지만, 여러분의 코드가 여러분의 통제
아래 머무른 채로 실행할 수 있도록 만들어졌습니다.

## 메모리 레이어

코드는 진실의 일부일 뿐입니다. 어떤 결정의 *이유(why)*, 잘못된 가정에 대한
정정, 어떤 파일에도 적혀 있지 않은 제약, 무언가가 지금처럼 된 히스토리 — 이런
암묵지는 보통 저장소가 아니라 사람들의 머릿속에 있습니다.

Platty는 이를 기록할 수 있게 합니다. `platty memory`로 **why / correction /
constraint / context** 종류의 자유 형식 노트를 소스 오브 트루스의 특정 문서,
EPIC, 또는 항목(item)에 앵커링합니다. 노트는 버전 관리되며(append-only 히스토리),
출처(누가 기록했는지, 신뢰도, 확정(confirmed)인지 제안(proposed)인지)를 함께
보존합니다.

핵심 이점은 이 지식이 **필요할 때 다시 돌아온다**는 점입니다: 앵커된 문서나
EPIC을 조회하면 메모리가 바로 옆에 함께 표시되고, Platty가 비즈니스 문서를
재생성할 때 기록된 메모리가 모델 컨텍스트로 다시 주입됩니다. 그래서 팀의 지식이
재생성을 거치며 사라지지 않고 유지되며, 결과물에 능동적으로 반영됩니다.

> **현재 동작:** 메모리는 **앵커**를 통해 다시 노출됩니다 — 메모리가 고정된
> 문서/EPIC을 열거나 재생성하면 메모리가 함께 따라옵니다. 자유 텍스트 검색은
> 현재 결과 옆에 메모리 존재 여부(개수)를 표시하며, 검색을 메모리 내용으로
> 직접 랭킹하는 기능은 로드맵에 있습니다.
