# MCP SDD PRD 형식

`prd.md`는 기획자·디자이너·QA가 읽는 한국어 문서다. 제품 기획 본문은 §0–§8에
두고, MCP 조사 상세·최신성·source parity·도구 호출·Self Review 근거는 **맨 아래
§9**에 둔다. 별도 영향도 파일은 만들지 않는다.

## Frontmatter

```yaml
---
id: "SPEC-<slug>-<YYYY-MM>"
type: "sdd-request"
status: "draft"
projectId: "<projectId>"
outputLanguage: "ko"
---
```

header에는 문서 식별과 승인 상태에 필요한 최소값만 둔다. 소스 커밋, 최신성,
근거 경계, impact revision/status, source parity, 조사 한계와 Self Review 기록은
모두 §9에 둔다.

## 작성 규칙

- `document_resolve`로 연결한 제품 문서와 `graph_trace` 결과의 상세는 §9에
  남긴다.
- 확인되지 않은 구현 경로를 제품 결정처럼 쓰지 않는다.
- §0~§8은 사용자·운영·정책 언어로 쓴다. 클래스명, 함수명, 파일·라인, code node,
  spec id, source parity, 검색·그래프 상태, 전체 enum/상태 전이 목록은 §9에 둔다.
  제품 규칙에 직접 필요한 최소한의 확인된 상태값만 한국어 의미를 먼저 설명하고 원래
  상태값을 괄호로 병기한다. 전체 enum이나 전이 목록은 옮기지 않는다.
- source parity, evidence confidence, 문서 작성 방식, MCP 도구 사용법은 제품 동작이 아니다.
  이를 `R-*`, `AC-*`, `D-*`, `H-*`로 만들지 않고 §9 또는 Self Review에 둔다.
- 미확인 내용은 `A-*` 가정 또는 `O-*` 미결 질문으로 남기고 `D-*` 확정 결정으로
  승격하지 않는다.
- §0의 `이번 리뷰에서 승인할 것`은 §3, §5, §7, §8의 기존 항목과 안정 ID만 요약한다.
  새 결정이나 새 근거를 이 행에서 만들지 않는다.
- `O-*`가 `closed`이면 해결 내용을 §6의 `D-*`로 확정하고 `해결 결정` 열에 연결한다.
  제품 질문을 기술 설계의 `DEC-*`로 대신 닫지 않는다.
- §8의 현재 기준값이나 목표가 확인되지 않았으면 값을 꾸며내지 않고 `A-*` 또는 `O-*`에
  연결한다. 성공 판정이 없는 측정 항목은 승인 가능한 가설로 표시하지 않는다.
- 제품 규칙은 `R-*`, 수용 기준은 `AC-*`, 결정은 `D-*`, 가정은 `A-*`, 미결 질문은
  `O-*`, 성공 가설은 `H-*`를 사용하고 문서 간에 다시 번호를 매기지 않는다.
- §9가 partial이면 본문의 범위 또는 검증에서 그 한계를 짧게 알린다.
