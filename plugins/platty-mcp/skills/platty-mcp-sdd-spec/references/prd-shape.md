# MCP SDD PRD 형식

`prd.md`는 기획자·디자이너·QA가 읽는 한국어 문서다. 제품 기획 본문은 §0–§8에
두고, MCP 조사 상세·최신성·source parity·도구 호출·Self Review 근거는 **맨 아래
§9**에 둔다. 별도 영향도 파일은 만들지 않는다.

## Frontmatter

```yaml
---
id: "SPEC-<slug>-<YYYY-MM>"
type: "sdd-request"
status: "draft"
projectId: "<projectId>"
outputLanguage: "ko"
---
```

header에는 문서 식별과 승인 상태에 필요한 최소값만 둔다. 소스 커밋, 최신성,
근거 경계, impact revision/status, source parity, 조사 한계와 Self Review 기록은
모두 §9에 둔다.

## 작성 규칙

- `document_spec_resolve`로 연결한 제품 문서와 `graph_trace` 결과의 상세는 §9에
  남긴다.
- 확인되지 않은 구현 경로를 제품 결정처럼 쓰지 않는다.
- §0~§8은 사용자·운영·정책 언어로 쓴다. 클래스명, 함수명, 파일·라인, code node,
  spec id, source parity, 검색·그래프 상태, 전체 enum/상태 전이 목록은 §9에 둔다.
  제품 규칙에 직접 필요한 최소한의 확인된 상태값만 한국어 의미를 먼저 설명하고 원래
  상태값을 괄호로 병기한다. 전체 enum이나 전이 목록은 옮기지 않는다.
- source parity, evidence confidence, 문서 작성 방식, MCP 도구 사용법은 제품 동작이 아니다.
  이를 `R-*`, `AC-*`, `D-*`, `H-*`로 만들지 않고 §9 또는 Self Review에 둔다.
- 미확인 내용은 `A-*` 가정 또는 `O-*` 미결 질문으로 남기고 `D-*` 확정 결정으로
  승격하지 않는다. 단, API, DTO, DB, index, migration, cache, query, 정렬 구현,
  tie-breaker, component, file, test, deployment, rollback처럼 같은 사용자 결과를 만드는
  기술 선택은 `O-*`가 아니라 `설계로 넘길 항목`에 기록한다.
- §0의 `이번 리뷰에서 승인할 것`은 §3, §5, §7, §8의 기존 항목과 안정 ID만 요약한다.
  새 결정이나 새 근거를 이 행에서 만들지 않는다.
- 설계 feasibility 피드백으로 승인된 제품 약속을 다시 열었을 때만
  `설계 피드백으로 조정한 약속` 행을 사용한다. 구현 클래스나 필드가 아니라
  사용자가 받게 될 결과의 전후 차이와 영향받는 안정 ID를 적고, 두 제품 파일의
  status를 `draft`로 되돌린다.
- `O-*`가 `closed`이면 해결 내용을 §6의 `D-*`로 확정하고 `해결 결정` 열에 연결한다.
  제품 질문을 기술 설계의 `DEC-*`로 대신 닫지 않는다.
- 추천안을 채택했다면 해당 `O-*`는 현재 revision에서 `closed`이고 `D-*`에 연결돼야
  한다. 나중에 별도 기획으로 재검토할 수 있다는 이유만으로 채택 결정을 open으로 두지
  않는다. 승인 핵심 결과를 바꿀 수 있는 `A-*`/`O-*`는 proven safe scope로 좁히거나
  `BLOCKING`으로 분류한다.
- `설계로 넘길 항목`은 제품 승인을 우회하는 장소가 아니다. 각 행은 보존할 사용자
  결과를 명시한다. 비용·보안·개인정보·데이터 손실·비가역 변경 또는 사용자 결과를
  바꾸는 선택은 기술 handoff로 숨기지 않고 제품 결정이나 명시적 기술 책임자 결정으로
  돌린다.
- §8의 현재 기준값이나 목표가 확인되지 않았으면 값을 꾸며내지 않고 `A-*` 또는 `O-*`에
  연결한다. 다만 사용자 약속이 현재 숫자에 의존하지 않으면 기준값 수집 기간과 수집 후
  적용할 정확한 pass/fail 규칙을 적고, 계측 구현은 설계 handoff로 보낼 수 있다.
  사용자에게 임의의 기준값을 만들도록 요구하지 않는다. 성공 판정이 없는 측정 항목은
  승인 가능한 가설로 표시하지 않는다.
- `baseline 대비 개선`, `측정 후 확정`, 단순 상향/하향은 성공 판정이 아니다. 각 `H-*`는
  숫자 목표 또는 비교 기간·guardrail·pass/fail 결과가 고정된 실행 가능한 결정 규칙을
  가져야 한다. 정하지 못하면 `O-*`를 열고 Self Review를 `NEEDS_WORK`로 둔다.
- 모든 `H-*`는 연결된 정상·예외 Given/When/Then과 함께 성립해야 한다. 통지는 발송
  시도, provider 수락, 실제 도달, 사용자 확인을 구분하고, 채널 실패 예외가 있는데
  무조건적인 미발송/미도달 0건을 약속하지 않는다.
- §9-8에는 모든 coverage limit를 아래 열로 분류한다. 제품 약속을 좌우하는 DESIGN/UCL,
  화면, 권한, 결제/write, 알림 근거가 빠졌으면 `BLOCKING`이다.

```markdown
| 한계 | 영향받는 제품·스토리 ID | 승인 영향 | 다음 정확한 확인 |
| --- | --- | --- | --- |
| <unread surface> | R-01 / AC-01 / US-01 / H-01 | BLOCKING | <exact MCP read> |
```

  `impactStatus: partial` 자체는 자동 실패가 아니지만, `BLOCKING` 행이 하나라도 있으면
  Self Review PASS와 제품 승인 요청을 금지한다. 모든 partial 행이 제품 결과에 영향을
  주지 않는다고 근거 있게 분류된 경우에만 `NON_BLOCKING`으로 둔다.
- 제품 규칙은 `R-*`, 수용 기준은 `AC-*`, 결정은 `D-*`, 가정은 `A-*`, 미결 질문은
  `O-*`, 성공 가설은 `H-*`, 설계 handoff는 `DH-*`를 사용하고 문서 간에 다시 번호를
  매기지 않는다.
- §9가 partial이면 본문의 범위 또는 검증에서 그 한계를 짧게 알린다.
